# 도미노 게임 추가 — 설계 문서

날짜: 2026-07-08

## 배경 / 목표

`mini-game-platform`(React + TypeScript + Vite, react-router)에 4번째 게임으로
표준 블록 도미노(더블식스, 28피스)를 추가한다. 싱글플레이(vs AI)와 온라인
멀티플레이(최대 4인, Firebase Realtime Database 기반) 두 모드를 지원한다.
기존 게임(2048, 사과게임, 테트리스)과 동일하게 `GameModule` 계약을 따르고,
`gameRegistry.ts`에 한 줄만 추가해 등록한다.

배포 대상은 GitHub Pages(정적 호스팅만) 이므로, 별도 서버 없이 클라이언트
SDK만으로 동작하는 실시간 동기화 방식이 필요하다.

## 파일 구조

```
src/games/domino/
  index.ts                 # GameModule export (id: "domino")
  Domino.tsx                # 최상위: 화면 상태머신 (menu → setup → playing → roundEnd/matchEnd)
  Domino.css                 # 플레이 화면 스타일 (초록 보드 + 흰/검 타일, 미니멀)
  DominoMenu.tsx              # 시작화면: 싱글/멀티 선택, 인원/닉네임/목표점수 입력 (이집트 테마)
  DominoMenu.css
  DominoTile.tsx               # 타일 렌더 컴포넌트 (pip 점 표시, 싱글/멀티 공용)
  engine/
    types.ts                    # Tile, MatchState, RoundState, Move 등 도미노 전용 타입
    deck.ts                       # 28피스 생성 + 셔플 + 분배
    rules.ts                       # canPlay/validMoves/applyMove(순수 리듀서)/점수 계산
    ai.ts                            # AI: 가능한 수 중 무작위 선택
  multiplayer/
    firebase.ts                      # Firebase 초기화 (env 변수 기반, 익명 인증)
    room.ts                            # 방 생성/참가/구독, move 이벤트 append
```

공통 파일 변경은 다음 두 곳, 각각 한 줄/한 블록 추가로 제한한다 (기존 세
게임이 쓴 패턴과 동일):
- `src/gameRegistry.ts` — `domino` import 및 배열에 추가
- `src/styles/tokens.css` — `--accent-domino` 계열 변수 추가(기존 줄 수정 없음)

`App.tsx`, `GameShell.tsx`, `useHighScore.ts`, 다른 게임 폴더는 수정하지
않는다.

## 라우팅 & 화면 흐름

App.tsx의 기존 라우팅(`/domino` 단일 경로)을 그대로 사용한다. 방 링크는
쿼리스트링으로 구현: `/domino?room=CODE`. 이 링크로 접속하면 자동으로
"참가" 화면으로 진입한다.

`Domino.tsx`는 내부적으로 화면 상태머신을 가진다:

```
menu ──┬─ single-setup ──────────────┐
       └─ multi (create/join) ───────┼─→ playing → roundEnd ─┬─→ playing(다음 라운드)
                                      │                        └─→ matchEnd
```

- `menu` / `single-setup` / `multi(create/join)` 단계는 `GameShell` 밖에서
  독자적인 이집트 테마 UI(`DominoMenu`)로 렌더링한다 (아직 점수가 없는
  단계라 GameShell 헤더가 맞지 않음).
- `playing` 단계만 `GameShell`로 감싼다. 턴 표시, 남은 타일 수, 참가자별
  누적 점수판은 `GameShell`의 `children` 영역 상단에 도미노 전용 미니멀
  바로 구현한다 (GameShell 자체는 수정하지 않음).

## 게임 규칙 엔진 (`engine/`)

프레임워크와 무관한 순수 함수로 구현해 싱글/멀티 양쪽에서 동일하게
재사용한다.

- **덱**: 0-0~6-6 총 28피스, 매치(첫 라운드) 또는 라운드마다 셔플.
- **분배**: 인원수와 무관하게 **각자 무조건 7피스**를 지급한다.
  - 2인: 14장 분배 / 14장 보유고
  - 3인: 21장 분배 / 7장 보유고
  - 4인: 28장 전부 분배 / 보유고 0장 (뽑을 타일이 없으면 곧바로 패스)
- **턴 진행**:
  1. `validMoves(hand, board)`로 낼 수 있는 타일을 계산한다.
  2. 없으면 보유고에서 한 장을 뽑아 확인한다. **낼 수 있는 타일이 나오거나
     보유고가 빌 때까지 반복**해서 뽑는다(표준 draw 규칙).
  3. 그래도 낼 수 없으면(보유고가 비었고 여전히 불가) 턴을 패스한다.
  4. 낼 수 있는 타일이 있으면(원래 손에 있었든, 방금 뽑았든) 플레이어가
     타일과 놓을 쪽(왼쪽/오른쪽 끝)을 선택해 놓는다. 필요 시 타일을
     뒤집어 자동으로 맞춘다.
- **라운드 종료**:
  - 누군가 손을 다 냄 → 그 라운드 승리
  - 아무도 못 내고 보유고도 빔(블록) → 손에 남은 핀 합이 가장 낮은 사람이
    승리. 동점이면 **턴 순서상 현재 딜러(라운드 시작 플레이어)로부터 가장
    가까운 사람**을 단독 승자로 정한다(결정적이고 구현이 간단한 타이브레이크).
- **점수**: 라운드 승자가 **나머지 전원의 남은 핀 합**을 획득해 누적 점수에
  더한다.
- **매치 종료 방식은 설정 화면에서 선택**:
  - **단판**: 라운드 1회로 게임 종료, 승자 발표
  - **목표점수**: 목표 점수(기본 100점, 설정 화면에서 조정 가능)에 처음
    도달한 사람이 나올 때까지 라운드를 반복. 라운드가 끝날 때마다 새로
    28피스를 셔플/재분배하고 다음 라운드를 시작한다. 직전 라운드 승자부터
    다음 라운드를 시작하는 것이 표준 관례이므로 이를 따른다.
- **AI**: 자기 턴이 되면 `validMoves` 중 무작위로 선택해 실행(뽑기/패스
  로직은 사람과 동일한 엔진 함수를 그대로 재사용). 사람이 보기 편하도록
  약 500ms 딜레이 후 자동 실행.

## 싱글플레이 상태 관리

`Domino.tsx` 내부에서 `useReducer`로 `MatchState` 전체를 로컬 관리한다.
사람 턴에는 UI로 타일/보드 끝을 선택하게 하고, AI 턴에는 엔진 함수를
자동으로 호출한다. 인원(2~4명 = 나 1명 + AI 1~3명)은 설정 화면에서 선택.

**개인 최고 기록**: `useHighScore("domino")`을 "내가 달성한 매치 최종
누적 점수의 최고 기록"으로 사용한다(GameShell 헤더의 SCORE/BEST에 표시).

## 멀티플레이 (Firebase Realtime Database)

- **선택 이유**: GitHub Pages 같은 정적 호스팅에서 별도 서버 배포 없이
  클라이언트 SDK만으로 동작. 방/턴 상태를 JSON 트리로 표현하기 쉬움.
  무료 Spark 플랜으로 캐주얼 사용량 충분.
- **동기화 모델**: `/rooms/{roomId}`에 방 메타데이터(참가자 목록, 목표
  점수, 현재 라운드 상태의 초기값 등)와 `moves`(이벤트 로그, append-only)를
  저장한다. 모든 클라이언트가 `onValue`로 구독하고, **동일한 순수 리듀서
  (`applyMove`)로 이벤트를 처음부터 재생**해 현재 상태를 계산한다. 이렇게
  하면 서버 로직 없이도 모든 클라이언트가 같은 상태로 수렴한다. 무브를
  만드는 클라이언트(현재 턴인 사람)만 `moves` 배열에 새 이벤트를
  append한다 — 동시에 두 명이 쓸 상황이 UI 상에서 원천적으로 없다(자기
  턴이 아니면 조작 불가).
- **손패 보호**: Firebase 익명 인증(anonymous auth)으로 클라이언트마다
  고유 uid를 발급받고, `hands/{uid}` 경로는 보안 규칙으로 본인만 읽을 수
  있게 제한한다.
- **알려진 한계 (사용자 승인 완료)**: 보유고(boneyard) 순서 자체는
  클라이언트가 계산해야 하므로, 개발자도구로 들여다보면 이론적으로 다른
  사람의 패를 유추할 수 있다. 서버 권위가 없는 완전 클라이언트 구조의
  본질적 한계이며, 친구끼리 하는 캐주얼 게임을 전제로 허용하기로 함.
- **방 생성/참가**:
  - 호스트가 "방 만들기" → 6자리 영숫자 코드 생성(혼동되는 문자 제외) →
    충돌 시 재생성 → `/domino?room=CODE` 링크 표시/복사.
  - 참가자는 그 링크로 접속 → 자동으로 참가 화면 진입 → 닉네임 입력 →
    합류(최대 4인, 이미 시작된 방이거나 4인 초과 시 안내 후 참가 불가).
  - 호스트가 "게임 시작"을 누르면 그 시점에 셔플/분배를 확정해 방 상태에
    기록한다.
- **재접속**: 로컬스토리지에 `(roomId, playerId)`를 저장해, 새로고침해도
  같은 자리로 복귀 가능하게 한다.
- **환경 설정**: Firebase 프로젝트를 사용자가 직접 콘솔에서 생성해야 한다
  (무료). 발급받은 설정값은 `.env`(커밋 제외, `.env.example` 제공)의
  `VITE_FIREBASE_*` 변수로 주입한다. 이 리포지토리 작업자는 실제 Firebase
  프로젝트를 만들 수 없으므로, 코드/보안 규칙 예시는 완성하되 실제 온라인
  플레이 테스트는 사용자가 프로젝트 생성 후 직접 확인해야 한다.

## 비주얼 디자인

- **보드**: 초록 펠트 배경(카지노/카페 테이블 느낌). 타일 체인은 가로
  스크롤 컨테이너에 배치(양 끝으로 확장), 더블(같은 숫자) 타일은 실물
  도미노처럼 90도 회전해서 표시.
- **타일**: 흰색 배경 + 검은 점(pip), 얇은 테두리 — 실물 도미노 느낌.
  `DominoTile` 컴포넌트로 통일.
- **시작화면(`DominoMenu`)**: 이집트 테마를 은은하게 반영 — 샌드스톤/
  파피루스 계열 색상, 앙크·스카라베·상형문자 모티프를 포인트 아이콘/
  구분선 정도로만 사용. 실제 플레이 화면(초록 보드/흰 타일)과는 확실히
  구분되는 별도 톤.
- **정보 노출**: 현재 턴, 남은 타일 수(본인 손패 개수 + 보유고 개수),
  참가자별 누적 점수판만 명확하게 노출. 불필요한 장식 배제.

## 검증 계획 / 환경 제약

- 이 작업 환경에는 64비트 Node가 설치되어 있지 않아(32비트 Node만 발견),
  `vite build`의 rolldown 네이티브 바인딩이 동작하지 않는다. 대신
  `tsc -b`(타입 체크)와 `oxlint`(린트)는 정상 동작함을 확인했다 — 구현
  중/후 이 두 가지로 검증하고, 최종 `npm run build`(번들링 포함)는
  사용자 환경에서 재확인을 요청한다.
- 싱글플레이(vs AI)는 브라우저에서 직접 플레이해 동작을 확인한다.
- 온라인 멀티플레이는 Firebase 프로젝트가 없어 실제 룸 생성/합류까지는
  이 세션에서 종단간 테스트가 불가능하다 — 로직/코드 완성 후 사용자가
  Firebase 프로젝트 생성 및 `.env` 설정 후 확인하도록 안내한다.
